# Детали программной реализации

> Этот документ описывает архитектуру, ключевые решения и логику работы программного комплекса моделирования автономной навигации НПА. Написан для упрощения подготовки текста дипломной работы.

---

## 1. Общая архитектура программы

Программа представляет собой модульный симулятор навигации необитаемого подводного аппарата (НПА). Она написана на Python 3.10+ и разбита на шесть независимых пакетов, каждый из которых отвечает за свою часть задачи:

| Пакет | Что делает |
|-------|-----------|
| `models/` | Математическая модель движения аппарата |
| `sensors/` | Модели бортовых датчиков (ИНС, DVL, компас, USBL) |
| `filters/` | Алгоритмы фильтрации (EKF, адаптивный EKF, Dead Reckoning) |
| `simulation/` | Генерация траекторий, синтетических данных, имитация сбоев, оркестрация прогона |
| `evaluation/` | Вычисление метрик точности и анализ робастности |
| `visualization/` | Построение графиков для анализа результатов |

Корневые файлы:
- `auv_params.py` — физические параметры аппарата и среды
- `config.py` — глобальная конфигурация эксперимента
- `main.py` — точка входа, запускает все сценарии

Такое разделение построено по принципу единственной ответственности (Single Responsibility): каждый модуль знает только о своей задаче и ничего не знает о внутреннем устройстве других модулей. Например, фильтр EKF не знает, какой именно датчик прислал измерение — он принимает универсальный формат (вектор измерений, матрица наблюдения H, ковариация шума R).

### Зависимости

Программа использует три основные библиотеки:
- **NumPy** — матричные операции, линейная алгебра (умножение матриц, обращение, вычисление Якобиана)
- **Matplotlib** — построение графиков траекторий, ошибок и адаптации R
- **Pytest** — автоматическое тестирование всех модулей

SciPy указана в зависимостях, но на данном этапе не используется напрямую — она зарезервирована для возможных расширений (например, UKF с сигма-точками).

## 2. Физические параметры аппарата (`auv_params.py`)

Все характеристики НПА и условия внешней среды собраны в одном месте — датаклассе `AUVParams`. Это сделано для того, чтобы изменение параметров аппарата автоматически пересчитывало все производные величины (ковариационные матрицы Q и P₀), и не приходилось вручную менять числа в нескольких файлах.

### Параметры аппарата

| Параметр | Значение | Смысл |
|----------|----------|-------|
| Масса | 50 кг | Масса аппарата |
| Макс. скорость V_max | 2.0 м/с | Предельная скорость хода |
| Постоянная времени τ | 3.0 с | Время разгона до 63% от V_max |
| Макс. угловая скорость ψ̇_max | 5°/с | Предельная скорость поворота |
| Шаг дискретизации Δt | 0.1 с | Интервал между шагами фильтра |

### Параметры среды

| Параметр | Значение | Смысл |
|----------|----------|-------|
| Плотность воды | 1025 кг/м³ | Морская вода |
| Температура | 20°C | Номинальная температура |
| Скорость звука | 1480 м/с | Для расчёта дальности USBL |

### Как из параметров аппарата вычисляется шум процесса Q

Ковариационная матрица шума процесса Q определяет, насколько мы «не доверяем» модели движения. Её нельзя задавать произвольно — она должна отражать реальные ограничения аппарата.

Логика вычисления:

1. **Максимальное изменение скорости за один шаг:** ΔV_max = (V_max / τ) × Δt = (2.0 / 3.0) × 0.1 ≈ 0.0667 м/с. Это максимум, на который скорость может измениться за 0.1 с, исходя из динамики разгона аппарата.

2. **Максимальное изменение курса за один шаг:** Δψ_max = ψ̇_max × Δt = 5°/с × 0.1 с = 0.5° ≈ 0.00873 рад.

3. **Стандартное отклонение шума (правило трёх сигм):** Предполагаем, что реальные отклонения от модели не превышают максимума с вероятностью 99.7%. Значит σ = ΔX_max / 3:
   - σ_wV = 0.0667 / 3 ≈ 0.0222 м/с
   - σ_wψ = 0.00873 / 3 ≈ 0.00291 рад

4. **Матрица Q:** Q = diag(0, 0, σ²_wV, σ²_wψ). Нули на первых двух позициях (координаты x и y) — не случайность. Координаты являются интегральными величинами: их неопределённость нарастает через модель движения (через скорость и курс), а не через прямой шум. Добавление шума к координатам привело бы к «двойному счёту» неопределённости.

### Как вычисляется начальная ковариация P₀

Матрица P₀ = diag(σ²_x₀, σ²_y₀, σ²_v₀, σ²_ψ₀) задаёт начальную неопределённость:
- Позиция: ±10 м (σ² = 100 м²) — аппарат примерно знает, где он находится
- Скорость: ±0.5 м/с (σ² = 0.25) — аппарат стоит на месте, но мы не уверены на 100%
- Курс: ±0.5° (σ² ≈ 7.6 × 10⁻⁵ рад²) — компас даёт начальное значение с небольшой погрешностью

### Шум DVL с учётом температуры

DVL (доплеровский лаг) имеет базовую погрешность σ_base = 0.02 м/с. При отклонении температуры воды от номинальных 20°C на ±5°C добавляется дополнительная погрешность σ_temp = 0.01 м/с. Суммарный шум вычисляется по формуле:

σ_DVL = √(σ²_base + σ²_temp) = √(0.0004 + 0.0001) ≈ 0.0224 м/с

Для наихудшего случая в симуляции используется округлённое значение σ_DVL = 0.03 м/с.

## 3. Модель движения (`models/kinematic_2d.py`)

Аппарат моделируется в горизонтальной плоскости (3 степени свободы: перемещение по X, по Y и поворот ψ). Вертикальная динамика (глубина) не рассматривается.

### Вектор состояния

Состояние аппарата описывается четырьмя числами: **x** = [x, y, v, ψ], где:
- x — координата «Север» (м)
- y — координата «Восток» (м)
- v — скорость (м/с)
- ψ — курс, угол рыскания (рад)

### Уравнение процесса (модель движения)

На каждом шаге состояние обновляется по формулам:
- x' = x + v × cos(ψ) × Δt
- y' = y + v × sin(ψ) × Δt
- v' = v + a × Δt
- ψ' = ψ + ω × Δt

Здесь **u** = [ω, a] — вектор управления: угловая скорость (рад/с) и линейное ускорение (м/с²). Управление формируется ИНС на основе показаний гироскопа и акселерометра.

### Матрица Якоби (линеаризация)

Поскольку уравнения нелинейны (содержат sin и cos), для расширенного фильтра Калмана (EKF) необходима линеаризация. Матрица Якоби F = ∂f/∂x вычисляется аналитически:

```
F = | 1  0  cos(ψ)·Δt  −v·sin(ψ)·Δt |
    | 0  1  sin(ψ)·Δt   v·cos(ψ)·Δt |
    | 0  0  1            0            |
    | 0  0  0            1            |
```

Аналитическое вычисление Якобиана (а не численное через конечные разности) было выбрано потому, что оно точнее и быстрее. Для нашей модели Якобиан имеет простой вид, и вывести его вручную несложно.

### Нормализация углов

Курс ψ нормализуется в диапазон [−π, π] после каждого шага. Без нормализации фильтр может «увидеть» разницу между 359° и 1° как 358°, хотя реальная разница всего 2°. Эта же нормализация применяется к инновации (невязке) при обновлении фильтра, если измерение относится к курсу.

### Почему выбрана кинематическая модель, а не динамическая

Динамическая модель (с учётом сил, моментов, гидродинамических коэффициентов) точнее, но требует знания множества параметров (коэффициенты сопротивления, инерция и т.д.), которые сложно измерить. Кинематическая модель проще, но достаточна для задачи фильтрации: шум процесса Q компенсирует неточности модели. Это стандартный подход в подводной навигации.

## 4. Датчики (`sensors/`)

### Общая архитектура датчиков

Все датчики наследуют абстрактный класс `BaseSensor`, который определяет единый интерфейс:
- Метод `measure(true_state, timestamp)` — возвращает зашумлённое измерение
- Свойство `get_H()` — матрица наблюдения (связывает измерение с вектором состояния)
- Свойство `R` — ковариационная матрица шума

Каждый датчик при вызове `measure()` возвращает объект `SensorMeasurement`, содержащий значение, матрицу H, ковариацию R, имя датчика и временну́ю метку. Это означает, что фильтр получает всю информацию об измерении в одном пакете и ему не нужно знать, от какого именно датчика оно пришло.

### Инерциальная навигационная система (ИНС)

ИНС стоит особняком — она не является датчиком с матрицей H, потому что её данные подаются не в шаг обновления (update), а в шаг предсказания (predict). ИНС содержит:
- Гироскоп — измеряет угловую скорость ω (шум σ = 0.01 рад/с)
- Акселерометр — измеряет ускорение a (шум σ = 0.05 м/с²)
- Частота: 10 Гц (на каждом шаге симуляции при Δt = 0.1 с)

ИНС формирует зашумлённый вектор управления [ω, a], который подаётся на вход модели движения в predict().

### DVL (доплеровский лаг)

- Измеряет: скорость v (индекс 2 вектора состояния)
- Матрица наблюдения: H = [0, 0, 1, 0] — выбирает только скорость
- Шум: σ = 0.03 м/с (с учётом температурной погрешности)
- Частота: 5 Гц

DVL — ключевой датчик для оценки скорости. Его потеря (например, при проходе над обрывом, когда нет «дна» для отражения луча) приводит к быстрому росту ошибки скорости и, как следствие, позиции.

### Магнитный компас

- Измеряет: курс ψ (индекс 3 вектора состояния)
- Матрица наблюдения: H = [0, 0, 0, 1] — выбирает только курс
- Шум: σ = 0.03 рад (≈ 1.7°)
- Частота: 2 Гц

Компас корректирует дрейф гироскопа ИНС. Без компаса курс постепенно «уплывает», что влечёт ошибки в позиции (аппарат «думает», что плывёт на север, а на самом деле отклонился).

### USBL (гидроакустическая система)

- Измеряет: координаты [x, y] (индексы 0 и 1 вектора состояния)
- Матрица наблюдения: H = [[1,0,0,0], [0,1,0,0]] — выбирает координаты
- Шум: σ = 1.0 м по каждой координате
- Частота: 0.5 Гц (одно измерение каждые 2 секунды)

USBL — единственный датчик, который напрямую измеряет координаты. Однако он работает редко (0.5 Гц) и с высоким шумом (1 м). Это означает, что между обновлениями USBL аппарат полагается только на ИНС и DVL.

### Моделирование частоты обновления датчиков

Каждый датчик имеет свою частоту обновления. Метод `is_available(timestamp, dt)` проверяет, должен ли датчик выдать измерение на данном шаге симуляции. Проверка реализована через «тики»: вычисляется ближайший тик (момент, кратный периоду датчика) и проверяется, попадает ли он в текущий шаг (timestamp − dt, timestamp].

Это важно, потому что разные датчики выдают данные с разной скоростью:
- ИНС: каждый шаг (10 Гц = каждые 0.1 с)
- DVL: каждый 2-й шаг (5 Гц = каждые 0.2 с)
- Компас: каждый 5-й шаг (2 Гц = каждые 0.5 с)
- USBL: каждый 20-й шаг (0.5 Гц = каждые 2 с)

## 5. Фильтры навигации (`filters/`)

### Общий интерфейс

Все фильтры наследуют абстрактный класс `BaseFilter`, который хранит:
- Модель движения (ссылка на `Kinematic2DModel`)
- Текущую оценку состояния **x**
- Текущую ковариационную матрицу P
- Матрицу шума процесса Q

Интерфейс содержит два абстрактных метода:
- `predict(control, dt)` — шаг предсказания
- `update(measurement, H, R)` — шаг обновления

Такой дизайн позволяет добавлять новые фильтры (например, UKF) без изменения остального кода.

### Классический EKF (`filters/ekf.py`)

Реализует стандартный двухэтапный алгоритм расширенного фильтра Калмана:

**Шаг предсказания (predict):**
1. Вычисляем новое состояние по модели движения: x⁻ = f(x, u)
2. Вычисляем Якобиан F = ∂f/∂x
3. Прогнозируем ковариацию: P⁻ = F·P·Fᵀ + Q

**Шаг обновления (update):**
1. Инновация (невязка): ν = z − H·x⁻ (разница между измерением и прогнозом)
2. Ковариация инновации: S = H·P⁻·Hᵀ + R
3. Усиление Калмана: K = P⁻·Hᵀ·S⁻¹ (определяет, насколько доверять измерению)
4. Обновление состояния: x⁺ = x⁻ + K·ν
5. Обновление ковариации по формуле Джозефа

**Почему используется формула Джозефа:**

Стандартная формула P = (I − K·H)·P может давать отрицательные числа на диагонали из-за накопления вычислительных ошибок. Формула Джозефа:

P⁺ = (I − K·H)·P⁻·(I − K·H)ᵀ + K·R·Kᵀ

численно устойчивее — она гарантирует, что P останется симметричной и положительно полуопределённой. Дополнительно после каждого шага выполняется принудительная симметризация: P = (P + Pᵀ) / 2.

**Обработка сингулярной матрицы S:**

При обращении матрицы S возможна числовая сингулярность (определитель близок к нулю). В этом случае используется псевдообратная матрица (через `np.linalg.pinv`), что позволяет фильтру продолжить работу.

### Адаптивный EKF (`filters/adaptive_ekf.py`)

Адаптивный EKF расширяет классический EKF механизмом автоматической подстройки матрицы шума измерений R. Шаг предсказания идентичен классическому EKF, а шаг обновления содержит дополнительную логику адаптации.

**Зачем нужна адаптация R:**

Классический EKF использует фиксированную матрицу R, которая задаётся один раз при инициализации. Но в реальности шум датчиков может меняться: DVL зашумляется при турбулентности, компас «плывёт» вблизи магнитных аномалий, USBL теряет точность при многолучевом распространении. Если R фиксирована, фильтр «не знает» об изменении условий и может расходиться.

**Алгоритм адаптации:**

1. Инновации (ν_k = z_k − H·x_k⁻) накапливаются в скользящем окне размера N = 20.

2. Вычисляется эмпирическая ковариация инноваций:
   C_ν = (1/N) × Σ(ν_k · ν_kᵀ)
   Это «фактический» разброс невязок — то, что наблюдаем в реальности.

3. Вычисляется теоретическая ковариация инноваций:
   S = H·P⁻·Hᵀ + R
   Это «ожидаемый» разброс невязок — то, что предсказывает фильтр.

4. Если C_ν > S — значит, фактический шум больше, чем мы думали, и R нужно увеличить. Если C_ν < S — R можно уменьшить. Обновление:
   R_new = R_old + α × (C_ν − S)
   где α = 0.1 — коэффициент адаптации (плавная подстройка).

5. Ограничения: чтобы R не вырождалась (→ 0) и не расходилась (→ ∞), диагональные элементы ограничиваются диапазоном [0.1 × R₀, 10 × R₀], где R₀ — начальная матрица R.

**Раздельные окна инноваций:**

Поскольку датчики имеют разные размерности измерений (DVL — 1D, компас — 1D, USBL — 2D), адаптивный EKF ведёт раздельные скользящие окна для каждой размерности. Это означает, что адаптация R для DVL не влияет на R для USBL — каждый датчик адаптируется независимо.

**История адаптации:**

Каждое обновление R сохраняется в списке `r_history` для последующей визуализации. Это позволяет построить график того, как фильтр менял свою «оценку шума» во времени — важный результат для раздела 4.3 диплома.

### Dead Reckoning (`filters/dead_reckoning.py`)

Dead Reckoning (счисление пути) — наивный метод навигации, который использует только ИНС (интегрирование показаний гироскопа и акселерометра) без коррекции по внешним датчикам.

Реализация: шаг predict() выполняется точно так же, как в EKF (та же модель движения, тот же Якобиан, то же обновление ковариации). Шаг update() намеренно пустой — он принимает измерения, но игнорирует их. Это сделано для совместимости с общим интерфейсом: Dead Reckoning проходит через тот же SimulationRunner, что и EKF, просто не корректируется.

**Зачем нужен Dead Reckoning:**

Dead Reckoning служит «нижней границей» качества навигации. Его включение в эксперименты наглядно показывает, что происходит без фильтрации: ошибка неограниченно растёт (десятки–сотни метров за 5 минут). Это обосновывает необходимость применения EKF.

## 6. Генерация траекторий (`simulation/trajectory.py`)

Программа поддерживает четыре типа эталонных траекторий:

### Круговая траектория (circle)

Аппарат движется с постоянной скоростью (1.5 м/с) по окружности радиусом 50 м. Угловая скорость ω = v / R = 0.03 рад/с. Ускорение равно нулю. Это базовый сценарий — простой, предсказуемый, удобный для первичной оценки фильтров.

### Прямолинейная траектория (straight)

Аппарат движется по прямой с постоянной скоростью и курсом. ω = 0, a = 0. Это простейший тест: если фильтр не работает на прямой — он не будет работать нигде.

### Восьмёрка (eight)

Траектория в форме лемнискаты — сложный маршрут с частой сменой курса. Скорость и курс непрерывно меняются. Это стресс-тест для фильтров: проверяет устойчивость при интенсивном маневрировании. Управляющие сигналы (ω, a) вычисляются по конечным разностям из аналитических координат.

### Синусоидальная траектория (sine)

Аппарат движется вдоль оси X с синусоидальным отклонением по Y. Используется для проверки работы при плавных изменениях курса.

---

## 7. Генерация данных (`simulation/data_generator.py`)

`DataGenerator` принимает эталонную траекторию и модели датчиков, и создаёт полный набор синтетических данных для прогона фильтра:

1. **Управления ИНС:** Для каждого шага берутся истинные ω и a из траектории, к ним добавляется гауссов шум гироскопа и акселерометра. Результат — зашумлённые управления, имитирующие реальные показания ИНС.

2. **Измерения датчиков:** Для каждого датчика на каждом шаге проверяется `is_available(t, dt)`. Если датчик доступен — генерируется измерение: H·x_true + шум. Если нет — пропускаем. Это даёт реалистичные наборы измерений с разными частотами.

Зерно генератора случайных чисел (seed = 42) фиксировано для воспроизводимости. При каждом запуске программы результаты будут одинаковыми.

---

## 8. Оркестрация прогона (`simulation/runner.py`)

`SimulationRunner` — это «дирижёр» симуляции. Он принимает набор фильтров и данные, и прогоняет каждый фильтр по всем шагам:

1. Для каждого шага k = 0, 1, ..., N−1:
   - Вызывает `filter.predict(control_k, dt)` для зашумлённого управления ИНС
   - Проверяет, есть ли измерения от датчиков на этом шаге (по временно́й метке)
   - Если есть — вызывает `filter.update(z, H, R)` для каждого доступного измерения
   - Сохраняет текущую оценку состояния и ковариацию

2. Каждый фильтр прогоняется **независимо** — у каждого своя копия начального состояния. Результат для каждого фильтра — массив оценённых состояний (N × 4) и ковариаций (N × 4 × 4).

Такая архитектура позволяет сравнивать произвольное количество фильтров на одних и тех же данных.

---

## 9. Имитация сбоев датчиков (`simulation/sensor_failure.py`)

Модуль моделирует отказы датчиков путём удаления измерений в заданных временных окнах.

### Как это работает

Сценарий сбоя — это тройка (имя_датчика, время_начала, время_конца). `SensorFailureSimulator` получает полные данные симуляции (со всеми измерениями) и «вырезает» из них измерения указанного датчика в заданном интервале. Исходные данные не изменяются — создаётся глубокая копия.

### Предопределённые сценарии

| Сценарий | Датчик | Окно | Моделируемая ситуация |
|----------|--------|------|----------------------|
| Потеря DVL | DVL | 100–130 с | Проход над обрывом: DVL теряет «дно» |
| Сбой компаса | Compass | 150–160 с | Магнитная аномалия |
| Комбинированный | DVL + Compass | 100–130 с + 150–160 с | Наихудший случай |

### Что видит фильтр при сбое

Когда датчик «отказывает», фильтр просто не получает от него измерений — шаг update() для этого датчика пропускается. Фильтр продолжает работать, опираясь на оставшиеся датчики. Это реалистично: в реальности отказ датчика означает отсутствие данных, а не некорректные данные.

## 10. Метрики точности (`evaluation/metrics.py`)

Для количественной оценки качества фильтров вычисляются следующие метрики:

| Метрика | Формула | Смысл |
|---------|---------|-------|
| RMSE позиции | √(mean(‖Δx, Δy‖²)) | Среднеквадратичная ошибка координат |
| RMSE скорости | √(mean((v_est − v_true)²)) | Среднеквадратичная ошибка скорости |
| RMSE курса | √(mean((ψ_est − ψ_true)²)) | С нормализацией углов |
| MAE | mean(‖Δx, Δy‖) | Средняя абсолютная ошибка |
| Max ошибка | max(‖Δx, Δy‖) | Наихудший момент |
| Средняя ошибка | mean(‖Δx, Δy‖) | Типичный уровень ошибки |

Ошибка курса нормализуется в [−π, π] перед вычислением, чтобы разница между −179° и +179° была 2°, а не 358°.

Функция `evaluate_simulation()` автоматически вычисляет все метрики для всех фильтров в результате симуляции. `print_metrics_table()` выводит результаты в виде таблицы.

## 11. Анализ робастности при сбоях (`evaluation/robustness.py`)

Этот модуль отвечает на вопрос: «Насколько хорошо фильтр справляется с отказом датчика и как быстро восстанавливается?»

### Вычисляемые метрики

Для каждого фильтра и каждого сбоя вычисляются:

1. **Baseline (базовая ошибка)** — медиана ошибки позиции до начала сбоя (в стабильном режиме, от 10 с до момента сбоя). Используется медиана, а не среднее, чтобы исключить влияние начального переходного процесса фильтра (в первые секунды ошибка может быть высокой из-за большого P₀).

2. **Максимальная ошибка во время сбоя** — «худший момент», когда аппарат навигировался без одного из датчиков.

3. **Средняя ошибка во время сбоя** — характеризует качество навигации в целом на протяжении всего отказа.

4. **Время восстановления** — сколько секунд после окончания сбоя нужно фильтру, чтобы ошибка вернулась к уровню 1.5 × baseline. Множитель 1.5 нужен, чтобы учесть стохастические колебания ошибки вокруг baseline. Если ошибка не вернулась к baseline до конца симуляции — recovery_time = NaN.

### Генерация текста для диплома

Функция `generate_robustness_text()` автоматически формирует абзацы на русском языке, которые можно вставить в диплом. Пример:

> «Во время отключения DVL (100–130 с) максимальная ошибка позиции составила: EKF — 2.10 м, Adaptive EKF — 1.80 м. Средняя ошибка в окне сбоя: EKF — 1.50 м, Adaptive EKF — 1.20 м. Время восстановления после окончания сбоя: EKF — 8.2 с, Adaptive EKF — 3.5 с. Таким образом, Adaptive EKF показал максимальную ошибку на 14% ниже, чем EKF.»

Текст также сохраняется в файл `output/robustness_analysis.txt`.

## 12. Визуализация (`visualization/plotter.py`)

Модуль строит четыре типа графиков:

### 1. Сравнение траекторий
Показывает на одном графике истинную траекторию (чёрная сплошная линия) и оценки каждого фильтра (EKF — синий пунктир, Adaptive EKF — красный штрихпунктир, Dead Reckoning — зелёная точечная). Оси — «Y (Восток)» по горизонтали и «X (Север)» по вертикали, масштаб одинаковый.

### 2. Ошибки по координатам X и Y
Два субграфика: ошибка по X и по Y во времени. Позволяют видеть, в каком направлении фильтр ошибается больше.

### 3. Норма ошибки позиции
График ‖Δx, Δy‖ во времени — наиболее информативный для общей оценки. Показывает, как ошибка меняется: стабилизируется ли она (фильтр работает) или растёт (Dead Reckoning).

### 4. Адаптация матрицы R
Специальный график для адаптивного EKF. По оси X — номер шага обновления, по оси Y — диагональные элементы адаптированной матрицы R. Отдельные линии для:
- DVL/Compass (1D-измерения) — одна линия
- USBL (2D-измерения) — две линии (R[x,x] и R[y,y])

Если линия R растёт, значит фильтр «меньше доверяет» этому датчику. Если R стабильна — шум стационарный.

### Цветовая схема

Цвета фиксированы для единообразия во всех графиках:
- Истинная траектория: чёрный
- EKF: синий
- Adaptive EKF: красный
- Dead Reckoning: зелёный

## 14. Ключевые архитектурные решения

### Принцип инверсии зависимостей (Dependency Inversion)

Фильтры зависят от абстракции `BaseMotionModel`, а не от конкретной `Kinematic2DModel`. Если потребуется заменить модель движения (например, на 3D-модель), фильтры менять не нужно — достаточно создать новый класс, наследующий `BaseMotionModel`.

Аналогично, `SimulationRunner` зависит от абстракции `BaseFilter`, а не от конкретных EKF или AdaptiveEKF. Он может прогонять любой фильтр, реализующий интерфейс predict/update.

### Принцип открытости/закрытости (Open/Closed)

Метод `_adapt_R()` в AdaptiveEKF может быть переопределён в подклассе для реализации другой стратегии адаптации — без изменения существующего кода.

Новые типы траекторий добавляются в `TrajectoryGenerator` путём написания нового метода `_generate_xxx()` и добавления его в словарь `generators`.

### Датаклассы для всех структур данных

Все данные (параметры, результаты, метрики) передаются через датаклассы (`@dataclass`), а не через словари или кортежи. Это обеспечивает:
- Явную типизацию (IDE подсказывает поля)
- Самодокументирование (каждое поле имеет имя и описание)
- Неизменяемость (не получится случайно добавить несуществующее поле)

### Неизменяемость входных данных

`SensorFailureSimulator.apply()` создаёт глубокую копию (deepcopy) входных данных перед модификацией. Это гарантирует, что одни и те же данные можно использовать для разных сценариев: один раз без сбоев, другой раз со сбоями.

### Фиксация случайности

Зерно генератора случайных чисел (seed = 42) задаётся один раз в начале. Это делает все эксперименты детерминированными: при повторном запуске результаты будут идентичными. Это важно для воспроизводимости результатов диплома.

## 17. Краткое описание работы программы от запуска до результата

1. Создаётся объект `AUVParams` с физическими параметрами аппарата. Из них автоматически вычисляются Q и P₀.

2. Создаются модели датчиков (ИНС, DVL, компас, USBL) с заданными шумами и частотами.

3. Для каждого сценария:
   - Генерируется эталонная траектория (аналитические формулы)
   - `DataGenerator` создаёт зашумлённые данные всех датчиков
   - (Опционально) `SensorFailureSimulator` удаляет измерения в окнах сбоев
   - Создаются свежие экземпляры фильтров (EKF, Adaptive EKF, DR)
   - `SimulationRunner` прогоняет каждый фильтр по всем шагам
   - Вычисляются метрики точности

4. Для сценария со сбоями дополнительно запускается анализ робастности: вычисляются max ошибка во время сбоя, recovery time, и автоматически генерируется текст для диплома.

5. Строятся графики для каждого сценария и выводится сводная таблица.

## 18. Словарь терминов

| Термин | Описание |
|--------|----------|
| EKF | Extended Kalman Filter — расширенный фильтр Калмана |
| Adaptive EKF | EKF с адаптацией матрицы R по инновациям |
| Dead Reckoning | Счисление пути — навигация только по ИНС |
| Инновация (ν) | Разница между измерением и прогнозом: ν = z − H·x⁻ |
| Ковариация (P) | Матрица, описывающая неопределённость оценки |
| Якобиан (F) | Матрица частных производных модели по состоянию |
| Q | Ковариация шума процесса (насколько «врёт» модель) |
| R | Ковариация шума измерений (насколько «врут» датчики) |
| K | Усиление Калмана (вес между прогнозом и измерением) |
| H | Матрица наблюдения (связь z = H·x + шум) |
| RMSE | Root Mean Square Error — среднеквадратичная ошибка |
| DVL | Doppler Velocity Log — доплеровский лаг |
| USBL | Ultra-Short BaseLine — гидроакустическое позиционирование |
| ИНС | Инерциальная навигационная система |
